# 셀사이드 리서치 위키 — 스키마

이 파일은 LLM 에이전트가 이 위키를 어떻게 유지·관리할지 정의하는 계약서다.
모든 ingest/query/lint 작업 전에 반드시 이 문서를 먼저 읽는다.

## 목적
국내외 증권사 셀사이드 리서치를 종목·섹터·애널리스트·테마 단위로 누적·구조화한다.
RAG처럼 매번 원문을 다시 검색하지 않고, 지식을 한 번 컴파일해 계속 최신 상태로 유지한다.

## 3계층 구조
- `raw/reports/` — 원본 리서치(불변). 캡처 후 절대 수정하지 않는다. 파일명: `YYYY-MM-DD_증권사_종목또는섹터.md`
- `raw/social/` — SNS·블로그 등 비(非)셀사이드 원본(불변). 파일명: `YYYY-MM-DD_플랫폼_계정.md` (예: `2026-06-06_네이버블로그_헨리의퀀트대학.md`). RSS/본문 캡처 + takeaway 보존.
- `wiki/` — LLM이 소유하는 마크다운. 사람은 읽기만 한다.
  - `wiki/macro/` — 거시지표 페이지 (환율·유가·VIX·공포탐욕·금리·CDS)
- `CLAUDE.md` — 이 스키마.
- `index.md` — 카테고리별 네비게이션. **`tools/build_index.py`가 각 페이지 프론트매터에서 자동 생성**하며 표는 `updated` 최신순(내림차순) 정렬. 직접 편집 금지(매일 재생성됨).
- `log.md` — 시간순 append-only 기록 (ingest/query/maintenance 추적, 날짜 프리픽스)

## 페이지 타입 (wiki/ 하위)

### 종목 페이지 (wiki/stocks/)
필수 프론트매터:
```
type: stock
ticker: 005930
name: 삼성전자
sector: [[반도체]]
created: YYYY-MM-DD
updated: YYYY-MM-DD
```
본문 섹션: 투자의견 컨센서스 추이 / 목표주가 히스토리(증권사·날짜·TP) / 핵심 투자포인트 / 리스크 / 최근 실적 / 상충하는 견해(애널리스트 간) / 관련 출처 링크

### 섹터 페이지 (wiki/sectors/)
type: sector. 본문: 섹터 전망 / 주요 종목 [[링크]] / 업황 사이클 / 정책·매크로 변수

### 애널리스트 페이지 (wiki/analysts/)
type: analyst. 본문: 소속 증권사 / 커버리지 종목 / 과거 콜의 적중 이력 / 성향(강세/약세 편향)
- **비(非)증권사 논객 확장**: SNS/블로그/유튜브 등 독립 논객도 이 타입으로 둔다. `affiliation`에 출처(예: 네이버 블로그·독립)를 적고, 셀사이드 애널리스트와 구분되도록 본문 첫 줄에 명시. 예: [[헨리의 퀀트대학]].

### 테마 페이지 (wiki/themes/)
type: theme. 예: [[HBM]], [[온디바이스AI]], [[밸류업]]. 본문: 정의 / 수혜·피해 종목 / 관련 리서치

### 거시 페이지 (wiki/macro/)
type: macro. 거시경제 지표를 지표별 1페이지로 누적.
필수 프론트매터:
```
type: macro
name: 원달러환율
category: 환율        # 환율/금리/원자재/변동성/신용
unit: KRW/USD
created: YYYY-MM-DD
updated: YYYY-MM-DD
```
본문 섹션: 최신 스냅샷 / 스냅샷 히스토리(날짜·값·출처, append-only) / 셀사이드 전망(증권사·날짜·전망레벨·기간·출처 — 환율·유가·금리·CDS만, VIX·공포탐욕지수는 생략) / 영향 관계(관련 [[종목]]·[[섹터]] 위키링크) / 관련 출처 링크
대상 지표: [[원달러환율]], [[유가]], [[VIX]], [[공포탐욕지수]], [[한국금리]], [[미국금리]], [[한국CDS]]

## 위키링크
모든 종목·섹터·애널리스트·테마 언급은 `[[ ]]`로 연결한다.

## 최신 요약 블록 (페이지 상단)
모든 종목·섹터·테마 페이지는 제목/소개 직후, 본문 섹션들보다 **위에** `## 📌 최신 요약 (YYYY-MM-DD)` 블록을 둔다.
- 가장 최근 ingest 기준 핵심 takeaway 3~6개(현재가/목표가 분포·최신 이벤트·핵심 리스크 등)를 불릿으로 요약. 출처는 본문에 이미 있으므로 요약엔 생략 가능.
- 이 블록은 매 ingest마다 **갱신(덮어쓰기)**한다. 원문 정보는 아래 본문에 그대로 누적되므로 정보 삭제가 아니다(요약은 포인터).
- 상세 시계열·이벤트·날짜 섹션("## YYYY-MM-DD 업데이트" 등)은 **최신 날짜가 위로 오는 내림차순**으로 정렬한다(봇이 아래에 append해도 COMPACT가 최신순으로 재정렬).
- 거시 페이지(`wiki/macro/`)는 기존 `## 최신 스냅샷` 줄이 이 역할을 겸하므로 별도 요약 블록을 두지 않는다.

## SNS·블로그 소스 규칙
SNS/블로그(X, 네이버 블로그, 유튜브 등)도 리포트와 동일하게 **원본→정제** 흐름을 따르되, 신뢰도가 다르므로 별도 규칙을 적용한다.
- **원본 보존**: 글을 `raw/social/`에 캡처(불변). RSS 목록 + 핵심 글 takeaway + 링크.
- **정제 후 반영**: 원문 덤프 금지. 핵심 주장만 추려 관련 [[종목]]/[[섹터]]/[[테마]]/[[거시]] 페이지에 반영하고, **증분 인사이트(기존에 없는 내용)** 위주로 적는다.
- **출처 라벨**: 셀사이드(`🌐외국계`/`🇰🇷국내`)와 구분되게 **`🐦SNS·계정명`**으로 표기하고 원본 raw 경로를 단다.
- **미확인/루머**: 검증 안 된 추측은 `⚠️미확인`으로 표기하고, 기존 견해와 충돌하면 "상충하는 견해" 섹션에 병기. 확정 사실처럼 쓰지 않는다.
- **계정 추적(선택)**: 고신호 계정은 [[애널리스트]] 타입(비증권사 논객)으로 페이지화. 남발 금지.
- **수집 팁**: 네이버 도메인은 직접 fetch가 막혀 RSS(`rss.blog.naver.com/{id}.xml`)도 프록시(allorigins/jina) 경유가 필요할 수 있다.
- **유튜브 자막 파이프라인**: `tools/yt_ingest.py`로 채널/영상 자막을 받아 `raw/social/`에 캡처한다 (`python tools/yt_ingest.py @핸들 --limit 5`). 사용법은 `tools/README.md`. 자막은 자동생성이 많아 `⚠️미확인` 처리. `raw/social/`도 사이트에 퍼블리시되어 탐색기에서 열람 가능하다. 위키에서 캡처를 참조할 땐 **위키링크**로 단다(예: `[[raw/social/2026-06-06_네이버블로그_헨리의퀀트대학|캡처]]`) — 코드(백틱) 표기는 클릭이 안 되니 지양.

## 워크플로

### INGEST (소스 추가)
1. `raw/reports/`(셀사이드) 및 `raw/social/`(SNS·블로그)의 새 파일을 읽는다. SNS 소스는 "SNS·블로그 소스 규칙"을 함께 따른다.
2. 핵심 takeaway를 사용자와 짧게 논의한다.
3. 관련 종목/섹터/애널리스트/테마 페이지를 **생성 또는 갱신**한다. 한 리포트가 보통 5~15개 페이지를 건드린다.
   - **건드린 모든 페이지는 프론트매터 `updated`를 해당 ingest 날짜로 반드시 갱신한다** (신규 생성 시 `created`도 기입). Quartz가 페이지 표시 날짜로 `updated`를 쓰므로, 본문만 추가하고 `updated`를 두면 사이트의 날짜가 실제 갱신보다 뒤처진다.
   - **상단 `## 📌 최신 요약` 블록을 이번 ingest 기준으로 갱신**한다(없으면 생성). 새 상세 내용은 본문 하단에 시간순 append. ("최신 요약 블록" 규칙 참조)
4. 목표주가·투자의견 변경은 히스토리 테이블에 **추가**(덮어쓰지 않음).
5. 기존 페이지 내용과 **상충하면** "상충하는 견해" 섹션에 날짜·출처와 함께 명시한다. 절대 임의로 한쪽을 지우지 않는다.
6. **거시 스냅샷**: 리포트 기준일의 거시지표 7종을 웹검색해 각 `wiki/macro/` 페이지 스냅샷 히스토리에 1행 append한다. **같은 날짜 행이 이미 있으면 건너뛴다(idempotent).** 값을 못 구하면 `N/A`로 두고 사유를 적는다(가짜 숫자 금지). 최신 스냅샷 줄과 `updated`도 갱신한다.
7. `index.md`와 `log.md`를 갱신한다.

### QUERY (질의)
1. wiki/ 페이지를 먼저 검색한다 (원문 raw는 보조).
2. 출처 링크와 함께 답한다.
3. 가치 있는 발견이면 새 페이지로 저장하거나 기존 페이지에 반영할지 제안한다.

### LINT (헬스체크, 주 1회 권장)
- 도구: `tools/lint.py`(읽기 전용 보고 — 깨진 링크/고아/type누락/데이터공백), `tools/fix_wiki.py`(일괄 수정 — 명명정규화·쓰레기 헤드라인 링크 해제·type 보강). 자동화: `lint.yml`(주간 보고), `index.yml`(매일 fix_wiki+인덱스 재생성).
다음을 점검·보고한다:
- 모순/불일치 (예: 두 애널리스트의 TP가 충돌하는데 미표시)
- 오래된 주장 (더 최신 리포트로 폐기된 내용)
- 고아 페이지 (들어오는 링크 없음)
- 누락 크로스레퍼런스 (본문에 언급됐으나 페이지·링크 없는 종목/테마)
- 데이터 공백 (조사 필요 항목)

### COMPACT (정기 압축, 주 2회 자동)
봇이 매일 append만 해 페이지에 중복이 쌓이므로, 강력 모델(DeepSeek V4 Pro, OpenRouter 경유)로 비대 페이지를 정기 압축한다.
- 도구: `tools/compact.py` — 임계(기본 150줄) 초과 `wiki/` 페이지를 표준형태(`📌 최신 요약 → 핵심/쟁점 → 상충하는 견해 → 타임라인 → 출처`)로 재작성. 자동화: `.github/workflows/compact.yml`(월·목).
- **보존 규칙**: 구별되는 사실·날짜·출처·상충 견해는 최소 1회 유지(중복만 병합). 원본은 `raw/` 불변이라 손실 없음. `updated` 갱신.
- macro 페이지는 스냅샷 시계열이라 제외. 사용법은 `tools/README.md`.

## 원칙
- raw는 절대 수정하지 않는다.
- 정보를 지우지 않는다. 폐기된 견해는 날짜와 함께 보존하고 "obsolete"로 표시한다.
- 모든 수치(TP, 투자의견, 실적)는 출처와 날짜를 동반한다.
- 재실행은 항상 안전(idempotent)해야 한다.
